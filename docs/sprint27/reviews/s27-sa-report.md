### [NOTE-002] [Info] Pre-existing unchecked shift width in depthwise leaf-index rebuild

- **Category**: A04:2021 Insecure Design / C++ UB (theoretical)
- **Location**: `catboost/mlx/tests/csv_train.cpp:1803` — `partBits |= (goRight << lvl);` inside `ComputeLeafIndicesDepthwise`.
- **Evidence**: `lvl` ranges `[0, depth)` where `depth == actualTreeDepth`, driven by `config.MaxDepth` which is populated via `std::atoi(argv[++i])` at line 184 with no upper-bound clamp. `goRight` is `uint32_t`; a shift by `lvl >= 32` is C++ undefined behavior.
- **Why this is Info, not a finding**: `ComputeLeafIndicesDepthwise` is only reached by the `csv_train` validation/test binary, not by the Python public API or any production entry point. Reaching `depth >= 32` would already crash or mis-allocate at line 3815 (`numLeaves = (1u << actualTreeDepth)`, which itself is UB at depth ≥ 32) before the new shift runs — so the new code does not widen the blast radius. The pattern predates S27; S27 did not introduce or worsen it.
- **Remediation** (optional, defense-in-depth for a future sprint — NOT required for S27 PR):
  - Clamp `config.MaxDepth` at CLI-parse time to a sane bound (CatBoost's documented max is 16; even 24 would be safe).
  - Add an early assert at the top of `ComputeLeafIndicesDepthwise`: `Y_ASSERT(depth < 32);`.
- **Verification**: `config.MaxDepth` is clamped at argv parse or an assert guards the function entry.

## Surface-by-surface findings

### Surface 1 — C++ changes in `catboost/mlx/tests/csv_train.cpp` (commit `fb7eb59b5f`, +33/−19)

Result: **PASS**.

- **`unordered_map<ui32, TObliviousSplitLevel>` hash-flood DoS**: not applicable. Keys (`splitBfsNodeIds[i]`) are generated by the in-process training loop (`csv_train.cpp:3660–3683`), not parsed from an external API payload. An attacker capable of controlling BFS node IDs already controls the training binary's internals. Map size is bounded by `splits.size()` ≤ `2^depth - 1` ≤ 2^16 − 1 = 65 535 at CatBoost's documented max depth. Std `unordered_map` with integer keys is acceptable here.
- **Integer-overflow / shift-width**: `partBits |= (goRight << lvl)` is safe at the supported depth range; pre-existing max-depth clamping gap covered in NOTE-002.
- **`fprintf + exit(1)` invariant guard (lines 1789–1794)**: error message is static text plus three integer fields (`nodeIdx`, `depth`, `splits.size()`). No pointer addresses, no buffer contents, no user-controlled strings. Information-disclosure-safe. Appropriate use of `exit(1)` for an internal invariant violation in a validation binary.
- **Memory safety**: the new code performs only (a) `unordered_map` insert/lookup, (b) bitwise ops on `uint32_t` locals, and (c) an in-bounds read `dataPtr[d * lineSize + ns.FeatureColumnIdx]`. The bounds expression is unchanged from the prior implementation. No new raw-pointer arithmetic introduced by the sprint, no new allocations beyond the map and the pre-existing `leafVec`.
