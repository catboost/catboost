# BUG-S18-001 Post-Mortem — L1a Initial Kernel: 6-Order-of-Magnitude Parity Failure

**Status**: Resolved — commit `19fa5ce6cc`  
**Detected**: S18-04 parity run on initial L1a kernel (commit `abc4c229f9`)  
**Fixed**: commit `19fa5ce6cc` (cooperative-broadcast accumulation + butterfly removal)  
**Reporter**: @qa-engineer (S18-04)

---

## Symptom

S18-04 parity run against the initial L1a kernel (commit `abc4c229f9`) failed all 18 configs by 4–20 million ULP — six orders of magnitude above the DEC-008 tolerance envelope (RMSE ulp≤4, Logloss ulp≤4, MultiClass ulp≤8).

Notably: the **100-run determinism fixture passed**. Every run of the broken kernel produced the same wrong answer, bit-for-bit.

The broken kernel reported a `histogram_ms` of ~14 ms on the gate config — a 51% reduction from baseline, which would have looked like a gate-clearing result had parity not been checked first.

---

## Two compounding structural flaws

The initial L1a kernel was ported from D1c with two independent errors that compounded multiplicatively.

### Flaw 1: 1/32 doc-inclusion rate

**What the code did.** The per-thread accumulation loop iterated over documents with a step size equal to the full threadgroup width (256 threads). Each thread in lane `l` processed documents at indices `l, l+256, l+512, ...`. Under the new `simdHist[g][bin]` layout, ownership of bin `bin` within SIMD group `g` is held by exactly one lane: lane `bin & 31`. If the thread processing document `d` is not the lane that owns bin `doc.bin`, the write was silently discarded (the stride-partition ownership check dropped it).

**Net effect.** In a 32-lane SIMD group, only the 1 lane that owns a given bin also happened to be the thread assigned to process documents at the right stride offset. Across an entire iteration, approximately 1/32 of all documents contributed to the histogram. The other 31/32 were dropped silently.

**Why it was invisible to determinism.** The selection of which 1/32 of documents contributed was fully deterministic given the fixed dispatch geometry. Same hardware, same shader, same grid — same wrong answer every time.

### Flaw 2: 32× butterfly amplification

**What the code did.** The D1c intra-SIMD butterfly block (`simd_shuffle_xor` xor 16/8/4/2/1) was inherited unchanged. In D1c, this butterfly reduces 32 per-lane private `privHist` values into a single sum across the SIMD group. In the L1a layout, `simdHist[g][bin]` is already a shared slot — all 32 lanes in SIMD group `g` see the same value for that bin. The butterfly read from `simd_shuffle_xor(simdHist[g][bin], offset)`, which returned the same `simdHist[g][bin]` value regardless of `offset` (every lane holds the same shared slot). Five rounds of butterfly accumulation produced: `value × 2^5 = value × 32`.

**Net effect.** Every accumulated partial was multiplied by 32 before contributing to the final per-bin sum.

---

## Why it almost looked right

The two flaws partially cancelled in a misleading way:
- Flaw 1 reduced the accumulated sum by ~32× (1/32 of documents included).
- Flaw 2 amplified the accumulated sum by ~32× (butterfly over shared slots).

Net error: `(1/32) × 32 = 1.0×` at the order-of-magnitude level. The broken kernel's histogram sums were roughly the right magnitude — off by a factor of ~5, not 1000. Quick sanity checks during development (eyeballing loss magnitudes, checking that training loss decreases) could pass because the scale was plausible. Only a rigorous parity comparison against the reference binary exposed the 6-order-of-magnitude ULP divergence.

---

## Root cause

The L1a kernel was produced by mechanically adapting the D1c kernel source to the new `simdHist` layout, without re-deriving the accumulation algebra from first principles.

D1c's reduction was designed for per-lane private partials: each lane holds a different value, and the butterfly correctly combines them. The key invariant of D1c's butterfly is: **each lane holds a distinct partial that needs to be summed with its SIMD peers**.

L1a's layout breaks that invariant. Under stride-partition with `simdHist[g][bin]` in threadgroup memory, a bin slot is already shared — all lanes in the SIMD group read the same physical location. Applying a butterfly designed for per-lane state to per-group state is algebraically incorrect.

The correct derivation for L1a: because ownership is one lane per bin, there are no partials to reduce within a SIMD group. The intra-SIMD butterfly has nothing to do and must be removed entirely.

## ASCII root cause diagram

```
D1c layout (correct butterfly application):
  lane 0: privHist[bin] = 2.1   ← distinct value per lane
  lane 1: privHist[bin] = 3.4
  lane 2: privHist[bin] = 1.7
  ...
  lane 31: privHist[bin] = 0.9
  → butterfly reduces 32 distinct values → correct bin sum

L1a broken layout (butterfly applied to shared slot):
  lane 0: simdHist[g][bin] = 7.8   ← all lanes see the same slot
  lane 1: simdHist[g][bin] = 7.8   (shared threadgroup memory)
  ...
  lane 31: simdHist[g][bin] = 7.8
  → butterfly reads 7.8 from all 32 lanes → result = 7.8 × 32 = 249.6  ✗

L1a fixed layout (butterfly removed, stride-partition ownership):
  lane (bin & 31) is the sole writer for bin within SIMD group g
  → simdHist[g][bin] already holds the full per-SIMD-group sum after accumulation
  → no intra-SIMD reduction needed
  → cross-SIMD 8-term linear fold (DEC-009) folds 8 SIMD groups → final bin sum  ✓
```

## Fix (commit `19fa5ce6cc`)

Two changes:

1. **Replaced per-thread stride accumulation with a cooperative 32-doc batch loop.** Each batch iteration broadcasts lane `src`'s (packed feature value, stat) pair to all 32 lanes via `simd_shuffle`. Only the bin-owner lane writes (`bin & 31 == lane_index_in_simdgroup`). Every document contributes exactly once with zero atomics.

2. **Removed the intra-SIMD D1c butterfly entirely.** `simdHist[g][bin]` holds the full per-SIMD-group sum after accumulation. The only reduction that runs is the cross-SIMD 8-term linear fold from DEC-009.

Reduction depth after fix: the 5-level intra-SIMD butterfly is gone. The 8-term linear cross-SIMD fold (DEC-009) remains. Effective reduction depth improves from γ_12 (S17) to γ_7 (S18), tightening the Higham bound from ~7.2e-7 to ~4.2e-7.

## Results on fixed kernel

- 108/108 parity checkpoints bit-exact (18 configs × 6 checkpoints). ULP = 0 across all loss types.
- 100/100 determinism runs bit-exact.
- Actual `histogram_ms` gate config: **9.56 ms** (-66.8%) — substantially better than the broken kernel's misleading 14 ms.

See `docs/sprint18/parity_results.md` (S18-04b) and `docs/sprint18/results.md` (S18-05b) for full tables.

## Lessons

**1. Do not port reduction patterns across layout changes without re-deriving algebraic correctness.** A butterfly designed for per-lane private partials is incorrect when applied to shared per-group slots. The structural invariant must be verified, not assumed to transfer.

**2. Determinism-PASS alone is not evidence of correctness.** The broken kernel was deterministic because its errors were geometry-determined, not data-ordering-dependent. A kernel that consistently computes the wrong thing will pass any determinism fixture. Parity against a reference implementation is a separate and stronger check.

**3. Parity-first, not perf-first, on any reduction rewrite.** The broken kernel reported a plausible-looking performance improvement. A perf-first review would have merged incorrect code. The S18-04 parity gate caught the bug before the perf result was taken seriously.

**4. Compounding flaws can partially cancel, making sanity checks misleading.** The 1/32 undercount and the 32× butterfly amplification nearly cancelled at the order-of-magnitude level. This made the broken kernel look plausibly correct on quick inspection. Only a rigorous ULP comparison revealed the true error.
